
**Findings**

1. Critical crash risk in the shared text-field layout code. `textLayoutOffsetInViewport` force-casts two render objects without null safety in text_scrollview.dart and text_scrollview.dart. If either context is unavailable during lifecycle churn, the frontend can hard-crash. The same unsafe pattern repeats nearby in that file, so this is not an isolated edge case.

2. High crash risk in iOS/text-selection drag handling. The drag handlers in editing_controls.dart and editing_controls.dart also assume render objects exist and cast them directly. That is a lifecycle-sensitive path, especially around overlays and selection handles, so it can fail during widget transitions.

3. Medium layout bug in the landing page word-cloud sizing. The cached breakpoint logic in landing_page.dart only recomputes after the viewport changes by more than 50 px, but the actual breakpoint is 900 px and the ratio is used at landing_page.dart. That means a resize crossing 900 px by less than 50 px can leave the cloud stuck in the wrong layout until the window moves further.

4. Medium accessibility gap across clickable shell and landing-page controls. Several interactive elements are still built as `GestureDetector`-only targets with no keyboard activation or full semantics, including the header logo in _page_header.dart, the theme toggle in _floating_controls.dart, the expandable project cards in landing_page.dart, and the social rows in landing_page.dart. On web, that leaves keyboard users and some assistive-technology flows under-supported.
